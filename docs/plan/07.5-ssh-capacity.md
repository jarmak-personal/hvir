# Phase 7.5 — SSH Capacity & Telemetry Multiplexing

**Read first:** [`00-overview.md`](00-overview.md); design.md ADR-006 (PTY supervisor,
resume, and `HarnessAdapter` telemetry), ADR-008 (projects/workspaces), and ADR-010
(`ProjectHost`, one logical host identity, and no installed remote server).

## Why this phase exists

Phase 7 hands-on testing restored several workspaces and terminals correctly, then
exposed the physical limit hidden by a single multiplexed SSH transport. OpenSSH commonly
allows ten shell/exec/subsystem sessions per network connection. hvir currently spends
that same budget on PTYs, one context follower per agent terminal, two active-project
watchers, SFTP, and short Git/filesystem commands. Serializing buffered execs preserves a
last control slot under modest load, but it is containment rather than a scaling model:
ten PTYs alone can exhaust the usual connection allowance.

The user's normal workload is 10+ terminals on one SSH machine across multiple projects
and worktrees. That is a supported workload, not an exceptional server-tuning case.

## Goal

Keep 12+ terminals live on one logical SSH host while Git, files, watches, context meters,
and session recovery remain available. Collapse per-terminal telemetry followers into
adapter-owned host hubs, and let one `SshHost` own a bounded pool of physical SSH
transports. The renderer, project registry, persisted terminal records, and host-qualified
paths continue to see exactly one host.

## Design guardrails

- Pooling stays entirely behind `SshHost` / `ProjectHost`. Never create one `SshHost` per
  project, workspace, worker, or terminal; those would duplicate logical host identity,
  trust, prompts, caches, and reconnect state.
- PTYs still spawn only through the PTY supervisor. Harness file conventions, filtering,
  and parsing stay behind `HarnessAdapter`.
- No server configuration requirement. A larger `MaxSessions` may improve capacity, but
  hvir must work with an ordinary unmodified SSH server and cannot assume the configured
  limit is discoverable.
- No persistent installed helper or remote daemon. A temporary process launched through
  `ProjectHost.execStream` is acceptable and must disappear when its channel closes.
- Authentication material is never persisted by this work. Reusable password/passphrase
  answers may be retained only in memory for the lifetime of the logical host connection;
  keyboard-interactive/2FA flows must remain explicit and must not create prompt storms.
- Channel pressure may delay freshness, never paint. Queueing, reconnect, telemetry
  parsing, and pool growth stay off the renderer thread.

## Tasks

### Decision record and capacity model

- [ ] Add an ADR-010 addendum that distinguishes one **logical `SshHost`** from its
      physical `ssh2.Client` transports. Record the role-aware pool and rejected
      alternatives: requiring a larger server `MaxSessions`, one host/connection per
      workspace, one connection per PTY, terminal-screen scraping, and an installed
      remote helper.
- [ ] Write down the channel accounting invariants in code and tests: PTY, buffered exec,
      streaming exec, and SFTP each consume a server session channel; the logical pool
      reserves control capacity before admitting terminal work.
- [ ] Add bounded internal diagnostics for transport role, occupancy, spillover, and
      channel-open refusal. Do not expose credentials, command contents, or a new settings
      dashboard; logs and test hooks are sufficient for this phase.

### Adapter-owned telemetry hubs

- [ ] Replace one follower per terminal with at most one live telemetry stream per
      `(host, adapter)` — currently one Codex hub and one Claude Code hub. Plain shells
      subscribe to neither.
- [ ] Keep the hub interface generic while each adapter owns session discovery, remote
      follower construction, record filtering, and parsing. No Codex/Claude path or event
      conventions may leak into `SshHost`, the PTY supervisor, or the renderer.
- [ ] Support dynamic subscribe/unsubscribe keyed by hvir PTY id plus exact harness
      session id. Debounce hub reconfiguration; a restart-on-subscription-change design is
      acceptable when bounded replay closes the event gap and duplicate snapshots are
      harmless.
- [ ] Use explicit bounded framing from the remote follower to main (session identity plus
      payload), safe across arbitrary SSH chunks and concurrent writers. Reject malformed,
      oversized, ambiguous, or cross-session records without stopping other subscribers.
- [ ] Cache the latest valid telemetry per PTY and rehydrate every live subscription after
      a hub/transport reconnect. A telemetry failure degrades only the affected meters to
      unavailable; it never exits or blocks a PTY.
- [ ] Ensure disposing the last subscription tears down the hub and all of its remote tail
      children/FIFOs. App quit, host disconnect, terminal exit, and failed identity
      discovery must leave no follower processes behind.

### Role-aware SSH transport pool

- [ ] Refactor the current physical-client state into an internal transport abstraction
      owned by one `SshHost`. Preserve one host id, trust record, connection state, SFTP
      cache, and renderer prompt flow across the pool.
- [ ] Keep control work (SFTP, watches, telemetry hubs, buffered Git/filesystem execs) on
      control transport(s) with reserved capacity. Allocate PTYs to terminal transports
      with a conservative soft channel budget below the normal server default.
- [ ] Grow the appropriate role pool lazily when its soft budget is full or the server
      refuses a channel. Retry/spill to another healthy transport without failing existing
      PTYs; cap total transports and surface a calm capacity error if the host refuses new
      authenticated connections.
- [ ] Open new physical connections serially so pool growth does not collide with SSH
      unauthenticated-connection limits. Share successful host-key trust and reusable
      in-memory authentication for the logical connection; cancellation aborts the whole
      pending growth operation cleanly.
- [ ] Pin every PTY to its transport for its lifetime. A failed terminal transport reports
      exits exactly once through the PTY supervisor; harness resume remains the recovery
      mechanism, with no claim of preserving a remote process through a network drop.
- [ ] Define logical connection state from the control plane. A failed auxiliary transport
      must not mark every project disconnected when the control transport is healthy;
      control failure retains the existing reconnect/backoff and renderer state contract.
- [ ] Close idle auxiliary transports after their PTYs and streams are gone (with a short
      anti-churn grace period), and close every transport/follower before app quit.

### Verification

- [ ] Add deterministic fake-`ssh2` tests with deliberately small per-client session
      limits. Prove admission, queueing, spillover, refusal recovery, exact-once exits,
      reconnect, disposal, and one logical connection-state stream.
- [ ] Add telemetry-hub tests with 10+ mixed subscriptions: at most one channel per
      adapter, correct session routing under split/coalesced chunks, bounded replay,
      malformed-frame isolation, unsubscribe cleanup, and reconnect rehydration.
- [ ] Extend the Electron smoke or a focused integration harness so multiple restored
      terminals coexist with Git Changes/History, SFTP reads, and watch refreshes while
      pool growth occurs off-renderer.
- [ ] Document and run a real-host protocol using at least two projects and three
      workspaces on one SSH alias. Include agent output churn, Git changes, worktree
      discovery, switching, quit/relaunch, disconnect/reconnect, and teardown checks.

## Acceptance criteria

- [ ] On an unmodified SSH server, 12 live terminals across at least two projects and
      three workspaces on the same host remain usable while Files, Git Changes, History,
      worktree discovery, and both watchers continue refreshing without channel-open
      failures.
- [ ] Ten Codex sessions use one Codex telemetry channel; ten Claude Code sessions use one
      Claude telemetry channel. Meters remain session-correct through compaction, workspace
      switches, hub restart, and terminal exit.
- [ ] Growing the pool never produces repeated password/passphrase prompts when reusable
      authentication is available in memory. Keyboard-interactive/2FA growth is serialized,
      clearly labeled, cancellable, and never loops modal prompts.
- [ ] Quit/relaunch and a deliberate SSH disconnect preserve the Phase 6/7 recovery
      contract: sessions are offered/restored per workspace, pool transports are rebuilt
      invisibly, and no orphaned remote telemetry followers remain.
- [ ] A server that refuses further physical connections shows one calm capacity error,
      keeps existing terminals alive, and recovers after capacity becomes available.
- [ ] The renderer stays responsive during 12-terminal output churn plus Git/watch load;
      automated verification, the real-host protocol, and the status table are updated.

## Non-goals

Changing SSH server configuration. A persistent remote agent/daemon. Preserving live
remote processes across a broken network transport. Cross-host pooling. Terminal
orchestration/queues. Broader harness analytics, cost history, or the parked v2 telemetry
viewer. UI polish unrelated to a calm capacity/reconnect error belongs to Phase 8.
