# Phase 7.5 — SSH Capacity & Telemetry Multiplexing

**Read first:** [`00-overview.md`](00-overview.md); design.md ADR-006 (PTY supervisor,
resume, and `HarnessAdapter` telemetry), ADR-008 (projects/workspaces), and ADR-010
(`ProjectHost`, one logical host identity, and no installed remote server).

## Why this phase exists

Phase 7 hands-on testing restored several workspaces and terminals correctly, then
exposed the physical limit hidden by a single multiplexed SSH transport. OpenSSH commonly
allows ten shell/exec/subsystem sessions per network connection. hvir currently spends
that same budget on PTYs, one context follower per agent terminal, two active-project
watchers, SFTP, and short Git/filesystem commands. Serializing buffered execs preserves a
last control slot under modest load, but it is containment rather than a scaling model:
ten PTYs alone can exhaust the usual connection allowance.

The user's normal workload is 10+ terminals on one SSH machine across multiple projects
and worktrees. That is a supported workload, not an exceptional server-tuning case.
Telemetry subscriptions change only when a harness PTY is identified, restored, or exits
— not when the user switches projects/workspaces. Steady-state churn is therefore low,
but relaunch can create a burst of subscriptions that the hub must reconcile without
restarting once per terminal.

## Goal

Keep 12+ terminals live on one logical SSH host while Git, files, watches, context meters,
and session recovery remain available. Collapse per-terminal telemetry followers into
adapter-owned host hubs, and let one `SshHost` own a bounded pool of physical SSH
transports. The renderer, project registry, persisted terminal records, and host-qualified
paths continue to see exactly one host.

## Design guardrails

- Pooling stays entirely behind `SshHost` / `ProjectHost`. Never create one `SshHost` per
  project, workspace, worker, or terminal; those would duplicate logical host identity,
  trust, prompts, caches, and reconnect state.
- PTYs still spawn only through the PTY supervisor. Harness file conventions, filtering,
  and parsing stay behind `HarnessAdapter`.
- No server configuration requirement. A larger `MaxSessions` may improve capacity, but
  hvir must work with an ordinary unmodified SSH server and cannot assume the configured
  limit is discoverable.
- No persistent installed helper or remote daemon. A temporary process launched through
  `ProjectHost.execStream` is acceptable and must disappear when its channel closes.
- Authentication material is never persisted by this work. Reusable password/passphrase
  answers may be retained only in memory for the lifetime of the logical host connection;
  keyboard-interactive/2FA flows must remain explicit and must not create prompt storms.
- Channel pressure may delay freshness, never paint. Queueing, reconnect, telemetry
  parsing, and pool growth stay off the renderer thread.
- Transport roles are strict. `ProjectHost.exec` is bounded, noninteractive control work;
  `execStream` owns explicit control services such as watches/telemetry; and `spawnPty`
  owns user-interactive commands. Work never silently spills between control and terminal
  transports.

## Tasks

### Decision record and capacity model

- [x] Add an ADR-010 addendum that distinguishes one **logical `SshHost`** from its
      physical `ssh2.Client` transports. Record the role-aware pool and rejected
      alternatives: requiring a larger server `MaxSessions`, one host/connection per
      workspace, one connection per PTY, terminal-screen scraping, and an installed
      remote helper.
- [x] Write down the channel accounting invariants in code and tests: PTY, buffered exec,
      streaming exec, and SFTP each consume a server session channel; the logical pool
      reserves control capacity before admitting terminal work.
- [x] Centralize the initial v1 policy as named, test-visible constants: at most **8
      physical transports per logical host**, a soft budget of **6 session channels on a
      control transport**, at most **2 control transports**, and **8 PTY channels on a
      terminal transport**. Open lazily and reuse before growing. This supports at least
      48 PTYs even when a second control transport is needed; revisit the constants only
      with real-host/profile evidence rather than guessing from a server's theoretical
      maximum.
- [x] Add bounded internal diagnostics for transport role, occupancy, spillover, and
      channel-open refusal. Do not expose credentials, command contents, or a new settings
      dashboard; logs and test hooks are sufficient for this phase.

### Adapter-owned telemetry hubs

- [x] Replace one follower per terminal with at most one live telemetry stream per
      `(host, adapter)` — currently one Codex hub and one Claude Code hub. Plain shells
      subscribe to neither.
- [x] Keep the hub interface generic while each adapter owns session discovery, remote
      follower construction, record filtering, and parsing. No Codex/Claude path or event
      conventions may leak into `SshHost`, the PTY supervisor, or the renderer.
- [x] Add bounded stdin writes to the existing `ExecStreamHandle` seam (local and SSH) so
      a hub can receive control messages without direct `ssh2` access or a new transport
      abstraction leaking into adapters.
- [x] Support dynamic subscribe/unsubscribe keyed by hvir PTY id plus exact harness
      session id through a **generation-based full-set reconcile protocol**. Coalesce a
      burst for 50 ms, send the newest complete subscription set with a monotonically
      increasing generation, and have both sides ignore older generations. Do not restart
      the hub for ordinary terminal churn; a new hub process is reserved for initial
      startup or transport/process failure.
- [x] Use explicit bounded framing from the remote follower to main (session identity plus
      hub epoch, subscription generation, and payload), safe across arbitrary SSH chunks
      and concurrent writers. Reject malformed, oversized, stale, ambiguous, or
      cross-session records without stopping other subscribers.
- [x] Cache the latest valid telemetry per PTY and rehydrate every live subscription after
      a hub/transport reconnect. A telemetry failure degrades only the affected meters to
      unavailable; it never exits or blocks a PTY.
- [x] Ensure disposing the last subscription tears down the hub and all of its remote tail
      children/FIFOs. App quit, host disconnect, terminal exit, and failed identity
      discovery must leave no follower processes behind.

### Role-aware SSH transport pool

- [x] Refactor the current physical-client state into an internal transport abstraction
      owned by one `SshHost`. Preserve one host id, trust record, connection state, SFTP
      cache, and renderer prompt flow across the pool.
- [x] Keep control work (SFTP, watches, telemetry hubs, buffered Git/filesystem execs) on
      control transport(s) with reserved capacity. Allocate PTYs to terminal transports
      with a conservative soft channel budget below the normal server default.
- [x] Make the role boundary executable, not descriptive: all hvir-issued `exec` calls
      remain cancellable/buffer-bounded control operations; long-lived watchers and
      telemetry are classified `execStream` control services; all interactive commands
      (including a user's rebase, cherry-pick, or long-running shell command) run inside a
      PTY on a terminal transport. Saturated control work grows/queues/degrades its own
      role and never borrows a terminal transport.
- [x] Grow the appropriate role pool lazily when its soft budget is full or the server
      refuses a channel. Retry/spill to another healthy transport without failing existing
      PTYs; cap total transports and surface a calm capacity error if the host refuses new
      authenticated connections.
- [x] Open new physical connections serially so pool growth does not collide with SSH
      unauthenticated-connection limits. Share successful host-key trust and reusable
      in-memory password/passphrase authentication for the logical connection; never cache
      keyboard-interactive/OTP answers. Cancellation aborts the whole pending growth
      operation cleanly.
- [x] Put pool authentication behind a finite per-transport state machine: one active
      prompt sequence per logical host, each offered auth method attempted at most once,
      at most 4 keyboard-interactive challenge rounds, the existing connection deadline,
      and no automatic pool-growth retry after a prompted failure/cancel. Late callbacks
      from a cancelled or superseded generation are ignored.
- [x] Pin every PTY to its transport for its lifetime. A failed terminal transport reports
      exits exactly once through the PTY supervisor; harness resume remains the recovery
      mechanism, with no claim of preserving a remote process through a network drop.
- [x] Define logical connection state from the control plane. A failed auxiliary transport
      must not mark every project disconnected when the control transport is healthy;
      control failure retains the existing reconnect/backoff and renderer state contract.
- [x] Reuse an idle auxiliary transport immediately. After its last PTY/stream closes,
      retain it for a concrete **5-minute idle grace period** before closing; cancel that
      timer on new admission. Keep the primary control transport until explicit disconnect
      or app quit, and close every auxiliary transport/follower immediately during either.
      The grace period is a centralized v1 constant, not a new setting absent evidence that
      users need to tune it.

### Review hardening

- [x] Make telemetry frame serialization owner-aware, bounded, stale-owner-healing, and
      adapter-independent. Killing a follower while it owns the write lock releases it
      promptly, force-cleans a stuck follower/tail, and cannot leave survivors in an
      unbounded remote retry loop.
- [x] Admit telemetry frames against each subscription's first reconciled generation,
      bounded by the current hub generation and epoch, so relaunch replay survives later
      subscription churn without accepting pre-admission or cross-session data.
- [x] Centralize hub registry eviction and follower/FIFO/tail cleanup in the generic hub;
      remove the unused duplicate telemetry cache and unused public teardown path. Read
      invariant follower identity from shell variables and generation with a shell builtin,
      leaving only payload encoding on the per-record external-command path.
- [x] Centralize channel-open retry/spill for buffered exec, streaming exec, PTY, and SFTP.
      Keep refusal diagnostics without permanently clamping a transport, cancel retry waits
      during operation/host teardown, admit four buffered control execs by default, and clear
      the prompted-growth block after a successful primary reauthentication.
- [x] Use Git's locale-independent usage exit status for legacy `worktree list` fallback,
      and avoid repository-context probes before that cheap compatibility decision.

### Real-host acceptance protocol (run still pending)

1. Use one ordinary SSH alias without changing `sshd_config`. Record the host OS and, when
   readable without privilege, `sshd -T | grep -i maxsessions`; this is evidence, never a
   prerequisite or instruction to raise the limit.
2. Register two repositories on that alias. In the first, create or reuse two worktrees so
   the session contains at least two projects and three workspaces total.
3. Start or restore 12 terminals across those workspaces, including ten sessions of one
   telemetry-capable adapter and at least one of the other adapter. Confirm every terminal
   accepts input and continues producing output.
4. While all terminals remain live, open Files and Git Changes/History in both projects,
   make tracked and untracked changes from terminals, create/delete a worktree, and confirm
   tree, changes, history, ignored status, and workspace badges refresh without any
   `Channel open failure` log.
5. Run sustained output in all terminals for 30 seconds while switching projects,
   workspaces, Files/Git, and terminal selection. Confirm context meters remain attached to
   the exact harness session through compaction/output updates.
6. Quit hvir normally, check the remote host has no `hvir-telemetry.*` temporary helper or
   orphaned `tail` owned by it, relaunch, and restore the offered sessions. Repeat the Git,
   file, and worktree checks before and after reconnecting the SSH alias.
7. Refuse one later connection in a controlled server/network fixture if available. Verify
   the single calm capacity error, existing terminal survival, and recovery on the next
   admission after capacity returns. Do not alter a shared production SSH server for this
   fault step.
8. Record terminal count, project/workspace count, adapter mix, any prompts, logs, teardown
   process check, and pass/fail beside the status table before checking the real-host item.

**Automated evidence (2026-07-14, macOS reference machine):** `npm run smoke:capacity`
mounted 12 real Ghostty terminal panes and ran 30 seconds of concurrent output plus
watch/Git rail churn. The raw distribution was emitted as `[smoke:capacity:raw]`; latency
was **18.7 ms p99 / 18.8 ms max** across 1,800 frame samples and 75 click
transitions. The same path then reloaded, restored all 12 terminal records, and required
Git Changes/History to be ready alongside them.
The focused pool suite separately proved two-control-transport capacity with 48 live PTYs,
refusal recovery, and logical-state isolation.

**Review-hardening evidence (2026-07-14):** all 245 tests pass, including a real local
helper whose follower is killed while owning the telemetry write lock, in-flight replay
across a newer reconcile, permanent-budget-clamp recovery, retry cancellation, four-way
default buffered-exec admission, post-reauthentication growth, and localized legacy Git
fallback. The production capacity smoke was rerun after these changes and passed at
**17.7 ms p99 / 17.8 ms max** across 1,800 frame samples and 75 click transitions, then
restored all 12 terminal records with Git Changes and History ready.

## Acceptance criteria

- [ ] On an unmodified SSH server, 12 live terminals across at least two projects and
      three workspaces on the same host remain usable while Files, Git Changes, History,
      worktree discovery, and both watchers continue refreshing without channel-open
      failures.
- [x] Ten Codex sessions use one Codex telemetry channel; ten Claude Code sessions use one
      Claude telemetry channel. Meters remain session-correct through compaction, workspace
      switches, hub restart, and terminal exit.
- [x] Growing the pool never produces repeated password/passphrase prompts when reusable
      authentication is available in memory. Keyboard-interactive/2FA growth is serialized,
      clearly labeled, cancellable, and never loops modal prompts.
- [ ] Quit/relaunch and a deliberate SSH disconnect preserve the Phase 6/7 recovery
      contract: sessions are offered/restored per workspace, pool transports are rebuilt
      invisibly, and no orphaned remote telemetry followers remain.
- [x] A server that refuses further physical connections shows one calm capacity error,
      keeps existing terminals alive, and recovers after capacity becomes available.
- [x] A failed terminal transport unsubscribes its affected telemetry sessions exactly
      once. Late events from the old hub epoch/generation are discarded, never appear in
      another terminal's viewer, and unaffected terminals continue receiving coherent
      telemetry without duplicates.
- [x] The renderer stays responsive during 12-terminal output churn plus Git/watch load;
      the automated 30-second probe meets the p99 <100 ms / max <500 ms budgets. Automated
      verification, the real-host protocol, and the status table are updated.

## Non-goals

Changing SSH server configuration. A persistent remote agent/daemon. Preserving live
remote processes across a broken network transport. Cross-host pooling. Terminal
orchestration/queues. Broader harness analytics, cost history, or the parked v2 telemetry
viewer. hvir-issued interactive Git lifecycle operations (rebase/cherry-pick/merge) remain
outside this view-first product; users may run them normally in a terminal. UI polish
unrelated to a calm capacity/reconnect error belongs to Phase 8.
